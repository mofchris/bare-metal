**D-019** | Stage B | _Restore merges rather than replaces; SRS state is never merged — it's rebuilt from the merged attempts._
Options considered: replace-on-restore (simple, but restoring a laptop backup on the phone would erase phone-only history — exactly the data loss the backup exists to prevent); merge (union). Merge rules: `attempts` union with identity = (questionId, at, sessionId) — append-only records are immutable, so equal keys are the same event, making restore idempotent; `lessonProgress` per lesson: "done" beats "in-progress", two "done"s keep the EARLIER completedAt (the historical first completion); `srsState` is not merged at all — it's derived state, so after merging attempts it is rebuilt by replay (one code path, no reconciliation bugs possible); `meta` stays local (installId identifies the device, lastExportAt is per-device). Tradeoff: replay cost on restore — linear in history, measured trivial (the 5000-record perf pin covers the same path). My question to Christopher: none.
Status: pending (ratify at Gate B).
