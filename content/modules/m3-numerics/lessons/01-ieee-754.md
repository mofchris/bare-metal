## What this lesson answers

Every number in a neural network is stored in a fixed number of bits. That
constraint decides which values can exist at all, which arithmetic is exact,
and which silently loses information.

This lesson works out what a float actually is, bit by bit. The rest of the
module is about what breaks when the bits run short, and none of it makes sense
without this.

## How is a real number packed into 32 bits?

A float is scientific notation implemented in hardware. In decimal you write
6.02 × 10²³, splitting a number into digits and a scale. A float does the same
in base two, because the hardware only stores zeros and ones.

The digits are called the **mantissa**. The scale is called the **exponent**. A
float's value is ± mantissa × 2^exponent, and both parts are packed into one
fixed budget of bits.

That fixed budget is the whole subject of this module. Two quantities compete
for it, and giving more to one always means giving less to the other.

For **float32** the 32 bits are divided like this:

| Field    | Bits | What it contributes                     |
| -------- | ---- | --------------------------------------- |
| sign     | 1    | plus or minus                           |
| exponent | 8    | the scale: which power of two           |
| mantissa | 23   | the digits: precision within that scale |

The exponent needs to express small numbers as well as large ones, so it must
be able to go negative. Its 8 bits hold no minus sign, so the format subtracts
a fixed amount instead. This is called a **bias**, and for float32 the bias is
127: whatever value the 8 bits hold, subtract 127 to get the real exponent. A
stored 128 means 2¹, a stored 126 means 2⁻¹, and one unsigned field covers both
directions.

Two consequences follow directly from the table.

**Range comes from the exponent.** With 8 bits the exponent covers roughly 10⁻³⁸
to 10³⁸, which sets how large and how small a float32 can be.

**Precision comes from the mantissa.** There are 23 stored bits plus one more
that is never stored. In base two, any nonzero number can be shifted so its
leading digit is 1, because 1 is the only nonzero digit available. That leading
1 is therefore guaranteed, and storing a guaranteed value would waste a bit, so
it is left out and assumed. Numbers written in this shifted form are called
**normalized**. The effective precision is 24 binary digits, which works out to
about **7 decimal digits**.

Keep range and precision separate in your head. The next lesson is entirely
about what happens when those two budgets are split differently.

## Why is 0.1 not 0.1?

Writing 1/10 in base two produces a fraction that repeats forever, in exactly
the way 1/3 repeats forever in base ten as 0.333…

A float has finitely many mantissa bits, so it cannot store an endlessly
repeating pattern. What gets stored is the nearest value the format can
represent, which is close to 0.1 but not equal to it.

Now the famous consequence. `0.1 + 0.2 == 0.3` is false in every language,
because the stored versions of 0.1 and 0.2 are each slightly off, their sum is
slightly off, and it lands somewhere other than the stored version of 0.3.

This is not a bug in any language. It is what a finite binary format is.

Two professional habits follow. Never compare floats with `==`. Compare instead
by asking whether the difference between them is smaller than a tolerance, and
choose that tolerance deliberately rather than copying a number from somewhere.

## Why does the gap between floats change size?

Here is a fact that surprises people: there are as many representable floats
between 1 and 2 as there are between 1024 and 2048.

The reason is in the structure. The mantissa provides a fixed number of steps,
and the exponent decides how much territory those steps have to cover. Between
1 and 2 they cover a distance of 1. Between 1024 and 2048 the same number of
steps covers a distance of 1024, so each step is 1024 times larger.

The distance from one float to the next has a name: the **unit in the last
place**, written ULP. The ULP is tiny near zero and enormous near 10³⁸.

**Machine epsilon** is the specific ULP just above 1.0. For float32 it is about
1.2 × 10⁻⁷.

Follow what machine epsilon means operationally. Add a number much smaller than
1.2 × 10⁻⁷ to 1.0, and the true answer falls between 1.0 and the next
representable float. Rounding sends it back to 1.0, so the addition had no
effect whatsoever. The small number vanished on arrival, silently, and lesson
04 shows what that does to a long sum.

## What happens on every single operation?

Almost every arithmetic result falls between two representable floats, so
almost every operation rounds.

The default rule is **round-to-nearest-even**: go to the closer of the two
neighbours, and when the value sits exactly halfway, choose the one whose last
bit is zero.

That tie-breaking rule exists for a reason worth knowing. If halfway cases
always rounded upward, a long sequence of them would push a running total
steadily upward, and the error would accumulate in one direction. Alternating
according to the last bit makes the halfway cases cancel out on average.

One rounding is harmless, introducing about 10⁻⁷ relative error, meaning error
measured as a fraction of the value rather than as an absolute amount. The
questions the rest of this module asks are what happens after a billion of them,
and what happens when the mantissa gets shorter.

## What are inf and NaN?

Two exponent patterns are reserved rather than meaning what the formula says:
all zeros and all ones.

They encode **±0**, and **±infinity**, which is where a result lands when it
grows too large for the format to hold. They also encode **NaN**, meaning
not-a-number, which is the result of undefined operations such as 0/0 or ∞−∞.

NaN has a deliberate property worth memorising: NaN is not equal to itself. So
`x != x` is true only when `x` is NaN, and that is how a NaN is detected.

The reserved patterns also encode **subnormals**. A subnormal is a very small
value below the normal range, made representable by giving up the implicit
leading 1. That purchase costs precision, so subnormals trade accuracy for a
gradual approach to zero rather than an abrupt fall to it.

You will meet inf and NaN again in lesson 04, where they appear as the visible
symptoms of a training run going wrong.

## Check your understanding

You add 1.0 and 0.00000001 (that is 10⁻⁸) in float32, and the result prints as
exactly 1.0.

Explain why, using the structure of float32, and say what you would change to
make the addition register. A correct answer notes that machine epsilon for
float32 is about 1.2 × 10⁻⁷, that 10⁻⁸ is more than ten times smaller than the
gap between 1.0 and the next representable float, so the true sum rounds back
to 1.0; and that using float64, whose longer mantissa gives a much smaller
epsilon, would let the addition register.
